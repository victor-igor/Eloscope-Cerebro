## [19/05/2026] ReabilitaCão — sync Google Calendar omite colorId quando inválido (vets com cor personalizada)

**Decisão:** worker `process-calendar-queue` passa a sanitizar o campo `colorId` antes de enviar pra Google Calendar API. Cores fora do range `'1'..'11'` (eventColors oficiais) são OMITIDAS do payload — evento herda automaticamente a cor do calendário do vet. Banco continua armazenando o valor original sem alteração.

**Why:**
1. Google Calendar API rejeita `colorId` fora de `1..11` com HTTP 400 "Invalid color id value" — após 3 retries, job marca `failed`
2. 3 vets do Eloscope salvam cor customizada no banco: Francine (`'22'` colorId calendar), Francielly (`'#c2c2c2'` hex), Danila (`'#9fe1e7'` hex) — total 6.542 agendamentos com cor inválida pra Google API

4. Não dá pra "consertar" cor no banco sem quebrar UX (frontend Eloscope renderiza cores como hoje); resolver no boundary de saída é o ponto certo
5. Padrão já existia em `google-auth/index.ts:617-624` — só faltava aplicar paridade no `process-calendar-queue/index.ts`

**Entregáveis:**
- `supabase/functions/process-calendar-queue/index.ts` — helper `sanitizeColorId()` + 5 callsites refatorados com spread condicional (omite campo quando undefined, não envia como null)
- Deploy via Supabase MCP: versão 31 ACTIVE (versão 30 saiu com `verify_jwt: true` por engano e quebrou o cron — corrigido pra `false` na mesma sessão)
- Cleanup: 1.531 jobs pending antigos da fila marcados como completed com nota explicativa (1.169 Danila 15/05 + 362 Angélica Abreu 18/05 — soft-deletes em massa do cleanup de órfãos)

## [18/05/2026] Política de credenciais locais — tokens nunca no git

**Decisão:** formalizar regra de segurança Eloscope — credenciais (tokens, API keys, senhas, chaves) **nunca** entram em arquivo versionado, mesmo em repo privado. Cada membro do time mantém suas próprias credenciais localmente (1Password / `.env` / config local MCP). Git só conhece metadados (URL, IDs públicos, propósito).

**Why:**
1. Repo privado hoje pode virar público amanhã (fork errado, share, vazamento de acesso) — token em git history é praticamente irreversível
2. Cada membro com token próprio reduz vetor de ataque e respeita compliance de OAuth/contratos
3. Lucas corrigiu explicitamente quando salvei 2 tokens uazapi (instância Lucas principal + LunaJ6 v2) em `seguranca/uazapi-instancias.md` — política formalizada na hora

**Entregáveis:**
- `seguranca/credenciais-locais.md` (novo) — guia operacional completo: hierarquia de armazenamento (1Password → env shell → .env → Keychain → MCP local), checklist pré-commit, lista do que pode/não pode ir pro git, fluxo de credencial nova, protocolo de vazamento
- `seguranca/permissoes.md` — header agora aponta pra `credenciais-locais.md`
- `.gitignore` — expandido com `*.token`, `*.secret`, `credenciais-*.local.md`, `secrets/`, `.secrets/`, `.mcp.json`
- `seguranca/uazapi-instancias.md` — limpo de tokens, só metadados (URLs, IDs, números, propósito)
- Auto-memory: `feedback_secrets_no_git.md` (regra persistente cross-session)

**Auditoria:** `git log -S "<token>"` confirmou que os tokens uazapi **não vazaram pro histórico** (arquivo era untracked, sobrescrito antes de qualquer `git add`). Rotação não necessária.

## [15/05/2026] MCP project-scoped — registrar via `claude mcp add --scope local`, não `.mcp.local.json`

**Decisão:** servidores MCP de escopo local-do-projeto são registrados via `claude mcp add --scope local <name> <command>`, gravando em `~/.claude.json` na seção do projeto. **Não existe** `.mcp.local.json` como formato válido — o Claude Code ignora.

**Why:**
1. O Claude Code lê apenas `.mcp.json` (project-shared, commitado) e o `~/.claude.json` (config user com seção por projeto). Arquivos com sufixo `.local.json` no projeto não são carregados.
2. Quando criamos `.mcp.local.json` achando que seria carregado, o MCP nunca conectou — `claude mcp list` não mostrava.
3. `.mcp.local.json` é o nome padrão do `gitignore` (linha existente no `.gitignore` desde o setup AIOX), o que reforça a confusão — mas não é um arquivo real lido por nada.

**Padrão correto:**
- **Compartilhado com time** (sem chaves): `.mcp.json` na raiz do projeto, commitado.
- **Local com chaves**: `claude mcp add --scope local <name> <command> --env KEY=value`. Grava em `~/.claude.json` (já fora do git).

**Owner:** Lucas
**Status:** aplicado — MCP `elobrain` registrado via local scope, conectado e funcional após restart

## [15/05/2026] `gbrain frontmatter generate --fix` — não rodar sobre `.aios-core/`, `.aiox-core/`, `.claude/`

**Decisão:** o comando `gbrain frontmatter generate --fix` é ferramenta agressiva — adiciona `title + type` em TODOS os `.md` sem frontmatter dentro do path passado. ## [16/05/2026] uazapi `/message/find` não filtra por chatid — limitação técnica documentada

**Decisão:** aceitar que o endpoint `POST /message/find` da uazapi.dev **não suporta filtro por chatid** (testadas 6 variantes: `{chatid: X}` top-level, `{where: {chatid: X}}`, `{where: {wa_chatid}}`, `{where: {chat_id}}`, `{where: {jid}}`, `{where: {sender}}`, `chatid` em querystring GET, `{where: {chatid: {eq: X}}}`). Endpoint retorna 200 OK mas ignora o filtro silenciosamente e devolve as últimas N mensagens globais (de grupos, na maioria).

**Workaround validado:** usar metadados do chat objeto retornados por `POST /chat/find`:
- `wa_lastMsgTimestamp` (epoch ms)
- `wa_lastMessageSender` (JID — compare com chatid pra saber se foi do dono do chat ou nosso)
- `wa_unreadCount`
- `wa_lastMessageType` (tipo, não conteúdo)

Suficiente pra classificar avanço (🟢🟡🔴⚪). **Não suficiente** pra preview de texto das últimas mensagens — esse recurso fica perdido até descobrir endpoint correto.

**Why importante:**
1. Skill `/whatsapp-pendencias` do plugin `claude-uazapi-elo` (cadastrada em 15/05) usa `{chatid: X}` top-level no payload — **provavelmente está retornando dados errados também**. Audit recomendado.
2. Sem preview, qualquer audit de leads tem menos contexto — Lucas precisa abrir WhatsApp pra ver a última conversa.
3. Pode ser limitação do plano uazapi.dev, ou bug, ou apenas API mal documentada. Vale investigar com suporte.
